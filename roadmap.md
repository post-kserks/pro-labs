# Архитектура и Roadmap проекта PixelDB

## 1. Обзор проекта
**PixelDB** — это учебная реляционная база данных, поддерживающая подмножество языка SQL. Проект разделен на две основные части:
*   **Сервер (Go):** Принимает TCP-подключения, парсит SQL-запросы, выполняет их и управляет данными, сохраняя их в формате JSON на диске.
*   **Клиент (C++17):** Состоит из разделяемой библиотеки (`libpixeldb`) и интерактивной консоли (`pixeldb-shell`), стилизованной под ретро-RPG.

---

## 2. Архитектура системы

Проект использует архитектуру клиент-сервер с взаимодействием по протоколу TCP. Формат обмена данными — текстовый (построчный JSON).

### 2.1 Протокол взаимодействия
Каждый запрос от клиента и ответ от сервера передается в виде одной строки JSON, завершающейся символом перевода строки (`\n`).
*   **Запрос (Request):** `{"id": "...", "query": "SELECT * FROM heroes;"}`
*   **Ответ (Response):** `{"id": "...", "status": "ok", "type": "rows", "columns": [...], "rows": [...], "affected": 0, "message": "..."}`

## 3. Детальное устройство сервера (Go)

Серверная часть расположена в директории `server/` и разбита на четкие слои обработки запроса:

### 3.1. Сетевой слой (`cmd/pixeldb-server/main.go`)
*   Поднимает TCP-слушатель (по умолчанию на порту 5432).
*   Для каждого нового подключения запускается отдельная горутина (`handleConnection`).
*   Использует `bufio.Scanner` для чтения потока по строкам.
*   Создает новую сессию `executor.Session` для каждого клиента, которая хранит состояние (например, выбранную текущую БД `USE database;`).

### 3.2. Лексический анализатор / Лексер (`internal/lexer`)
*   **Файл:** `lexer.go`
*   **Роль:** Превращает сырую строку SQL в последовательность токенов (`Token`).
*   **Особенности:** Написан вручную (ручной конечный автомат). Поддерживает ключевые слова (SELECT, INSERT, WHERE и др.), типы данных, строковые (`'...'`), числовые и булевы литералы, а также операторы (`=`, `!=`, `<`, `>=`, и т.д.).

### 3.3. Синтаксический анализатор / Парсер (`internal/parser`)
*   **Файлы:** `parser.go`, `ast.go`
*   **Роль:** Реализует рекурсивный спуск (recursive descent parser). Принимает токены и строит абстрактное синтаксическое дерево (AST).
*   **Абстракции:** 
    *   `Statement`: Базовый интерфейс для SQL-выражений (DDL: Create/Drop Database/Table, Use; DML: Select, Insert, Update, Delete).
    *   `Expression`: Интерфейс для дерева выражений `WHERE` (BinaryExpr, AndExpr, OrExpr, NotExpr, Value, ColumnRef).

### 3.4. Исполнитель / Executor (`internal/executor`)
*   **Файлы:** `executor.go`, `session.go`, `commands.go`, `eval.go`
*   **Роль:** Паттерн Command. Берет AST-дерево и вызывает соответствующие методы у движка хранилища (Storage Engine).
*   `CommandFactory` маршрутизирует AST узлы в конкретные команды (например, `SelectCommand`, `InsertCommand`).
*   **Оценка выражений (`eval.go`):** Содержит логику рекурсивного вычисления `WHERE` выражения (сравнение типов, приведение к float64 для чисел, логические операции) для каждой строки таблицы.

### 3.5. Подсистема хранения / Storage (`internal/storage`)
*   **Файлы:** `storage.go` (интерфейсы), `file_storage.go` (реализация)
*   **Роль:** Управление данными на жестком диске.
*   **Структура файлов:**
    *   `data/databases/<db_name>/_meta.json` — метаданные БД.
    *   `data/databases/<db_name>/<table_name>/_schema.json` — схема таблицы (типы колонок).
    *   `data/databases/<db_name>/<table_name>/_data.json` — данные таблицы (массив массивов).
*   **Многопоточность (Concurrency):**
    *   В `FileStorageEngine` используется `sync.RWMutex` для изоляции на уровне всей БД (`globalMu`) при DDL-операциях.
    *   Используются блокировки на уровне отдельных таблиц (`tableLocks`), что позволяет конкурентно читать/писать в разные таблицы или конкурентно читать одну таблицу (RWMutex).
*   **Атомарность записи:** Данные пишутся во временный файл `.tmp`, который затем атомарно переименовывается в целевой `os.Rename`.

## 4. Детальное устройство клиента (C++)

Клиентская часть расположена в `client/` и собирается с помощью CMake.

### 4.1. Библиотека `libpixeldb` (`client/lib/`)
*   **`Connection`:** Класс, инкапсулирующий создание POSIX-сокета, подключение, формирование JSON-запроса, отправку и чтение ответа по разделителю `\n`.
*   **`Result`:** Структура данных, куда парсится ответ от сервера. 
*   *Зависимости:* Для парсинга JSON, по-видимому, используется легковесная библиотека или самописный парсер (`json_utils.cpp`).

### 4.2. Интерактивная оболочка `pixeldb-shell` (`client/shell/`)
*   **Файлы:** `main.cpp`, `renderer.cpp`
*   **Роль:** REPL (Read-Eval-Print Loop). Считывает ввод пользователя, отправляет его через `libpixeldb`, получает `Result` и красиво отрисовывает таблицы в консоли (псевдографика ASCII).

## 5. Существующие зависимости и узкие места
1.  **Типы данных и приведения:** Вся математика при сравнении внутри `eval.go` приводится к `float64`, что может стать проблемой точности для больших `int64`.
2.  **Загрузка таблиц в память:** Метод `SelectRows`, `UpdateRows`, `DeleteRows` целиком считывают `_data.json` в память, преобразуют, фильтруют, а при изменении целиком пишут обратно. При увеличении размера таблицы это приведет к Out-of-Memory (OOM) и чудовищным задержкам.
3.  **Отсутствие индексов:** Поиск по `WHERE` (Scan) работает за `O(N)`, проходясь по каждой строке.
4.  **Сетевой протокол:** Отправка сырого JSON текстом очень неэффективна по размеру и скорости сериализации.
5.  **Транзакции:** Полностью отсутствуют транзакции (ACID), кроме защиты от гонок потоков на уровне мьютексов.

## 6. Roadmap для архитектора и разработчиков

Этот план направлен на превращение PixelDB из простого учебного проекта в надежную, масштабируемую in-memory или disk-based СУБД.

### ЭТАП 1: Устранение критических проблем памяти и типов (Performance & Correctness)
*   [ ] **Переход на Streaming (Итераторы):**
    *   *Задача:* В слое `storage` заменить `SelectRows() ([]Row, error)` на паттерн Iterator (или Cursor).
    *   *Эффект:* `executor` сможет читать JSON потоково (например, через `json.Decoder`), применять `WHERE` и отдавать результат по чанкам, не загружая всю базу в RAM.
*   [ ] **Строгая типизация в Evaluator:**
    *   *Задача:* Избавиться от слепого каста к `float64` в `eval.go` для чисел. Использовать раздельное сравнение для `int64` и `float64` для сохранения точности.

### ЭТАП 2: Улучшение формата хранения (Storage Layer)
*   [ ] **Переход с JSON на бинарный формат:**
    *   *Задача:* Отказаться от `_data.json` в пользу собственного бинарного формата со страничной организацией (Page-based storage, например, по 4KB). 
    *   *Эффект:* Это обеспечит быстрый O(1) доступ к конкретной строке без необходимости парсить текстовый файл.
*   [ ] **Write-Ahead Log (WAL):**
    *   *Задача:* Реализовать механизм логгирования перед изменением файлов (для обеспечения долговечности 'D' в ACID).

### ЭТАП 3: Оптимизация запросов и Индексы
*   [ ] **B-Tree Индексы:**
    *   *Задача:* Добавить создание индексов `CREATE INDEX`. В слое `storage` реализовать B-Tree дерево, указывающее на смещение строк в бинарном файле данных.
*   [ ] **Простейший Query Optimizer:**
    *   *Задача:* Научить `SelectCommand` анализировать AST-дерево `WHERE`. Если есть условие `id = 5`, оптимизатор должен использовать индекс, а не делать Full Table Scan.

### ЭТАП 4: Сеть и Клиентские драйвера
*   [ ] **Бинарный протокол (вместо JSON):**
    *   *Задача:* Заменить JSON поверх TCP на бинарный протокол (например, Protobuf или кастомный протокол, похожий на PostgreSQL Wire Protocol). Это снизит нагрузку на CPU (сериализацию) и уменьшит трафик.
*   [ ] **Пул соединений в `libpixeldb` (C++):**
    *   *Задача:* Сделать `Connection` потокобезопасным или реализовать `ConnectionPool` для использования в многопоточных C++ приложениях.

### ЭТАП 5: Транзакции и блокировки
*   [ ] **Переход от Table-level lock к Row-level lock:**
    *   *Задача:* Текущая блокировка `sync.RWMutex` лочит всю таблицу при `UPDATE`/`INSERT`. Переписать движок под MVCC (Multi-Version Concurrency Control) или блокировки на уровне строк/страниц.
*   [ ] **Синтаксис BEGIN, COMMIT, ROLLBACK:**
    *   *Задача:* Поддержать сессии транзакций на уровне Executor.

Этот roadmap предоставит команде четкий вектор по эволюции проекта. Архитектору следует начать с рефакторинга слоя `storage` на итераторы (ЭТАП 1), так как это самая острая архитектурная проблема при росте объемов данных.
