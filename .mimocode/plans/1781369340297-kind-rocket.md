## Phase 1: Type-Safe Row Comparison (quick win, low risk)

### Проблема
5 мест используют `fmt.Sprintf("%v", a[i]) != fmt.Sprintf("%v", b[i])` — медленно, небезопасно для типов, дублируется.

### Решение
Создать `valuesEqual(a, b storage.Value) bool` в `executor/eval.go` (или новом файле `executor/compare.go`):
```go
func valuesEqual(a, b storage.Value) bool {
    if a == nil && b == nil { return true }
    if a == nil || b == nil { return false }
    // Try numeric comparison
    if af, aok := toFloat(a); aok {
        if bf, bok := toFloat(b); bok { return af == bf }
    }
    // String comparison
    if as, ok := a.(string); ok {
        if bs, ok := b.(string); ok { return as == bs }
    }
    // Bool comparison
    if ab, ok := a.(bool); ok {
        if bb, ok := b.(bool); ok { return ab == bb }
    }
    // Fallback
    return fmt.Sprintf("%v", a) == fmt.Sprintf("%v", b)
}
```

### Файлы для изменения
- **Новый**: `server/internal/executor/compare.go` — `valuesEqual` + `rowsEqual` (перенести из commands_tx.go)
- `server/internal/executor/commands_dml.go:299` — заменить fmt.Sprintf pair
- `server/internal/executor/commands_new.go:315` — удалить compareRows, использовать valuesEqual
- `server/internal/executor/commands_tx.go:275,323` — удалить rowsEqual, использовать valuesEqual
- `server/internal/executor/aggregates.go:28` — оставить как есть (это key hashing, не comparison)

## Phase 2: Plan Cache for Prepared Statements (medium complexity)

### Проблема
PrepareStatement хранит только AST. Каждый EXECUTE пересоздаёт Command через CommandFactory. Нет кэша планов.

### Решение
Добавить `PlanCache` в `executor/executor.go`:
```go
type CachedPlan struct {
    stmt       parser.Statement
    cmd        Command
    createdAt  time.Time
}

type PlanCache struct {
    mu    sync.RWMutex
    plans map[string]*CachedPlan  // key: normalized SQL hash
    maxSize int
}
```

Интеграция в `commands_tx.go`:
- При EXECUTE: сначала искать в PlanCache по хэшу bound statement
- Если нашли — использовать кэшированный Command
- Если нет — создать через CommandFactory, сохранить в кэш
- PlanCache ограничен по размеру (LRU eviction)

### Файлы
- **Новый**: `server/internal/executor/plan_cache.go`
- `server/internal/executor/commands_tx.go` — ExecutePreparedCommand использует PlanCache
- `server/internal/executor/session.go` — добавить PlanCache в Session

## Phase 3: GiST Index — Real Implementation (high complexity)

### Проблема
GiST — prototype с O(n) linear scan. Нужна реальная R-tree или MBR (Minimum Bounding Rectangle) структура.

### Решение
Реализовать простую R-tree:
```
GiSTIndex
  ├── node: { MBR, children: []node }  — внутренний узел
  ├── node: { MBR, entries: []entry }   — лист
  └── entry: { Key, Pos, Min, Max }
```

Алгоритмы:
- **Insert**: ChooseSubtree → вставить в лист, split если переполнен
- **Delete**: mark-and-sweep (пометить удалённым, потом compact)
- **Search**: рекурсивный traverse по MBR overlap
- **Range/Overlap**: аналогично Search с interval comparison

### Файлы
- `server/internal/index/gist_index.go` — полная переработка
- **Новый**: `server/internal/index/rtree.go` — R-tree node structure
- `server/internal/index/gist_index_test.go` — **новый** тестовый файл

## Phase 4: WAL for Schema Changes (high complexity, critical for crash safety)

### Проблема
Два TODO в page_engine.go:
1. `writeSchemaLocked` (строка 582) — не пишет WAL
2. `rewriteTable` (строка 1419) — не пишет WAL при ALTER TABLE

### Решение
Добавить WAL op types для schema operations:
```go
const (
    OpSchemaWrite  byte = 0x60  // schema file write
    OpRewriteBegin byte = 0x61  //开始 rewrite table
    OpRewriteData  byte = 0x62  // rewrite data record
    OpRewriteCommit byte = 0x63 // rewrite complete
)
```

Изменения в `page_engine.go`:
1. `writeSchemaLocked`: перед `os.WriteFile` → `wal.Append(OpSchemaWrite, payload)`
2. `rewriteTable`: 
   - Перед началом: `wal.Append(OpRewriteBegin, ...)`
   - После каждой порции данных: `wal.Append(OpRewriteData, ...)`
   - После завершения: `wal.Append(OpRewriteCommit, ...)`
3. Recovery: если OpRewriteBegin есть но OpRewriteCommit нет → откатить (восстановить старый heap)

### Файлы
- `server/internal/wal/wal.go` — добавить новые Op constants
- `server/internal/storage/page_engine.go` — writeSchemaLocked + rewriteTable
- `server/internal/storage/page_engine.go` — recovery logic для rewrite

## Phase 5: Objects inside Storage Engine (high complexity)

### Проблема
Views, triggers, functions, procedures хранятся как raw JSON файлы через `os.WriteFile` (commands_ddl.go:559,623,687,801). Не защищены WAL, не транзакционны.

### Решение
Создать виртуальную таблицу `_system` в каждой БД для хранения объектов:
```
CREATE TABLE _system (
    object_type TEXT,    -- 'view', 'trigger', 'function', 'procedure'
    object_name TEXT,
    definition  TEXT     -- JSON payload
)
```

Альтернатива (проще): хранить объекты в catalog JSON с WAL-защитой:
1. Каждый DDL объект пишется через `storage.InsertRows("_system", ...)` 
2. Storage engine автоматически WAL-логирует через существующий механизм
3. При recovery — объекты восстанавливаются из WAL

### Файлы
- `server/internal/executor/commands_ddl.go` — все CREATE/DROP VIEW/TRIGGER/FUNCTION/PROCEDURE
- `server/internal/storage/page_engine.go` — начальная инициализация _system таблицы
- `server/internal/storage/file_storage.go` — начальная инициализация _system таблицы
