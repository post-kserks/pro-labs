# ТЗ: Устранение критических проблем VaultDB

## Контекст

Проведён код-ревью проекта VaultDB (Go-сервер + React/TS клиент). Обнаружено ~85 проблем в 4 модулях. Данное ТЗ охватывает устранение критических и высоких проблем, а также доведение заглушек до рабочего состояния.

---

## Блок 1: Транзакционная целостность (КРИТИЧЕСКИЙ)

### 1.1 Доведение undo-функций до рабочего состояния

**Проблема:** `undoInsert` удаляет ВСЕ строки таблицы вместо вставленных. `undoUpdate` и `undoDelete` — заглушки, возвращают `nil`. Откат транзакций полностью нерабочий.

**Файлы:**
- `server/internal/executor/commands_tx.go` (undo-функции, строки 160-199)
- `server/internal/executor/commands_dml.go` (DML-выполнение)
- `server/internal/txmanager/manager.go` (структура `PendingOp`)

**Требования:**

#### 1.1.1 `undoInsert`
- Текущее поведение: читает ВСЕ строки, удаляет все по индексам 0..N.
- Требуемое поведение: удалить только те строки, которые были вставлены данной транзакцией.
- Подход: В `op.Payload` (`*parser.InsertStatement`) содержатся `Rows [][]parser.Expression` и `Columns []string`. Необходимо:
  1. Вычислить вставляемые строки через `buildRows` (как в `InsertCommand.executeImmediate`).
  2. Прочитать текущие строки таблицы.
  3. Найти совпадения по значениям (или по `op.Row` если сохранять при apply).
  4. Удалить только найденные строки через `Storage.DeleteRows`.

#### 1.1.2 `undoUpdate`
- Текущее поведение: заглушка, возвращает `nil`.
- Требуемое поведение: восстановить старые значения изменённых строк.
- Подход:
  1. В `PendingOp.OldRow` уже сохраняется снимок всех строк (`commands_dml.go:390-396`), но это неэффективно и ненадёжно.
  2. **Изменение в `commands_dml.go`:** При буферизации UPDATE в `PendingOp.Row` сохранять `[]storage.Row` только затронутых строк (matching rows по WHERE) + их индексы.
  3. В `undoUpdate`: для каждой строки из `op.Row` вызвать `Storage.UpdateRows` с восстановлением старых значений.

#### 1.1.3 `undoDelete`
- Текущее поведение: заглушка, возвращает `nil`.
- Требуемое поведение: вставить обратно удалённые строки.
- Проблема: при буферизации DELETE (`commands_dml.go:576-588`) данные удалённых строк не сохраняются.
- Подход:
  1. **Изменение в `commands_dml.go`:** При буферизации DELETE прочитать строки, совпадающие с WHERE, сохранить их в `op.Row` (`[]storage.Row`).
  2. В `undoDelete`: вставить сохранённые строки через `Storage.InsertRows`.

### 1.2 Исправление `undoAppliedOps` — порядок отмены

**Файл:** `server/internal/executor/commands_tx.go:138-158`

- Убедиться, что откат идёт в обратном порядке applied ops.
- При ошибке откату логировать и продолжать (best-effort), но помечать транзакцию как повреждённую.

### 1.3 Сохранение WAL-записей при DML-буферизации

**Проблема:** Логические WAL-операции (`OpInsert`, `OpUpdate`, `OpDelete`) определены как константы, но никогда не записываются при DML. Записывается только `OpCommit`.

**Файл:** `server/internal/executor/commands_tx.go`

**Требование:** При буферизации каждой DML-операции добавлять соответствующую WAL-запись с полезной нагрузкой (DB, Table, данные строк). Это необходимо для восстановления после сбоя.

## Блок 2: Storage Engine (КРИТИЧЕСКИЙ)

### 2.1 Исправление `BufferPool.Unpin`

**Файл:** `server/internal/storage/buffer_pool.go:87-106`

**Проблема:** `Unpin(bufIdx, dirty)` игнорирует `bufIdx`, отpinning первую попавшуюся закреплённую страницу.

**Требование:** Удалить метод `Unpin(bufIdx, dirty)`. Использовать `UnpinPage(pid PageID, dirty bool)` (строка 109), который корректно работает через `cache` map.

### 2.2 Исправление `evict()` — бесконечный цикл

**Файл:** `server/internal/storage/buffer_pool.go:126-142`

**Проблема:** При `pinCnt > 0` на LRU-tail цикл `continue` возвращается к тому же элементу.

**Требование:** Заменить `continue` на `elem = elem.Prev()` для перехода к следующему (более свежему) элементу. Если все закреплены — вернуть `false`.

### 2.3 Исправление `RLock` с мутацией

**Файл:** `server/internal/storage/buffer_pool.go:145-176`

**Проблема:** `FlushAll` и `FlushDirtyPagesUpToLSN` используют `RLock`, но мутируют `entry.dirty = false`.

**Требование:** Заменить на `Lock()`/`Unlock()` (write lock).

### 2.4 `InvalidateTable` — проверка `pinCnt`

**Файл:** `server/internal/storage/buffer_pool.go:179-197`

**Проблема:** Принудительно удаляет закреплённые страницы.

**Требование:** Перед удалением проверять `entry.pinCnt > 0`. Если закреплена — пропускать или возвращать ошибку.

### 2.5 Подключение `BufferPool` к `PageStorageEngine`

**Файл:** `server/internal/storage/page_engine.go:404-417`

**Проблема:** `getPage`/`unpinPage` — dead code. `BufferPool` выделен (строка 82) но не используется.

**Требование:**
1. `getPage` должен использовать `e.bufPool.FetchPage(pid, hf)` вместо прямого `hf.ReadPage`.
2. `unpinPage` должен делегировать в `e.bufPool.UnpinPage(pid, dirty)`.
3. Все вызовы `getPage` должны сопровождаться вызовом `unpinPage` после использования.

### 2.6 `doCheckpoint` — блокировка `e.mu`

**Файл:** `server/internal/storage/page_engine.go:353-385`

**Проблема:** Итерация `e.tables` без мьютекса — паника при конкурентном `CreateTable`/`DropTable`.

**Требование:** `doCheckpoint` должен захватить `e.mu.Lock()` перед итерацией `e.tables` и сохранением каталога.

### 2.7 `HeapFile` — race `ReadPage`/`WritePage` с `Close()`

**Файл:** `server/internal/storage/heap/heapfile.go:84-121`

**Проблема:** `getSegment` возвращает `*os.File` и отпускает мьютекс. `Close()` может закрыть файл до завершения I/O.

**Требование:** Ввести refcount для сегментов: `getSegment` инкрементирует, `putSegment` (новый метод) декрементирует. `Close()` ждёт все refcount == 0.

### 2.8 `mutateRows` — устранение двойного скана

**Файл:** `server/internal/storage/page_engine.go:949-1032`

**Проблема:** Два полных скана таблицы — один для WAL, один для мутации.

**Требование:** Объединить оба прохода в один: записывать WAL-позиции и мутировать страницы в рамках одного `scanTuples`.

## Блок 3: WAL-durability (ВЫСОКИЙ)

### 3.1 `Checkpoint()` — crash-safe truncate

**Файл:** `server/internal/wal/wal.go:126-140`

**Проблема:** `Truncate(0) + Sync` не атомарно.

**Требование:** Заменить на:
1. Записать чекпоинт-маркер (`OpCheckpoint`) в конец WAL.
2. При восстановлении пропускать записи до последнего чекпоинта.
3. (Опционально) Периодическая компактация: записать post-checkpoint записи в temp-файл, sync, atomic rename.

### 3.2 Убрать per-write `Sync()`

**Файл:** `server/internal/wal/wal.go:193-198, 211-216`

**Проблема:** `file.Sync()` на каждую запись — катастрофа для производительности.

**Требование:** Убрать `Sync()` из `appendBytesLocked` и `appendBytesLockedWithTx`. Sync выполняется в `Flush()` (перед checkpoint) и `Close()`. При необходимости синхронного коммита — добавить опциональный параметр `sync bool`.

## Блок 4: Безопасность HTTP (ВЫСОКИЙ)

### 4.1 HTTP Server Timeouts

**Файл:** `server/internal/httpserver/server.go:82-89`

**Требование:** Добавить таймауты:
```go
ReadHeaderTimeout: 5 * time.Second,
ReadTimeout:       15 * time.Second,
WriteTimeout:      30 * time.Second,
IdleTimeout:       120 * time.Second,
```
Для SSE-эндпоинта `/api/live` — использовать `http.ResponseController` или отдельный сервер без `WriteTimeout`.

### 4.2 Ограничение размера HTTP-body

**Файл:** `server/internal/httpserver/server.go:180, 651`

**Требование:** В каждом обработчике, читающем body, добавить:
```go
r.Body = http.MaxBytesReader(w, r.Body, s.cfg.MaxRequestSizeBytes)
```
Обработать `http.MaxBytesError` → 413 ответ.

### 4.3 Подключение Rate Limiter

**Файлы:** `server/internal/httpserver/ratelimit.go`, `server/internal/httpserver/server.go`

**Проблема:** Rate limiter определён но не подключён к middleware.

**Требование:**
1. Инстанцировать `NewRateLimiter` в `New()` или `Start()`.
2. Обернуть `apiMux()` через `rl.Middleware(...)`.
3. Добавить конфигурацию rate limit (requests/sec, burst) в `ServerConfig`.

### 4.4 Исправление rate limiter

**Файл:** `server/internal/httpserver/ratelimit.go`

Требования:
- **X-Forwarded-For:** Использовать только `r.RemoteAddr`. Если за прокси — добавить конфигурацию trusted proxies и брать первый IP из цепочки.
- **Unbounded map:** Заменить `map[string]*tokenBucket` на LRU-кэш с лимитом (например, `github.com/hashicorp/golang-lru/v2`).
- **Cleanup goroutine:** Добавить публичный метод `Stop()` и вызывать его при graceful shutdown.

### 4.5 CORS — ограничение origins

**Файл:** `server/internal/httpserver/server.go:407`

**Проблема:** `Access-Control-Allow-Origin: *` с авторизованными запросами.

**Требование:**
- Добавить конфигурацию `AllowedOrigins []string`.
- Валидировать `Origin` заголовок против allowlist.
- Возвращать `Access-Control-Allow-Origin` только для совпадений.
- Добавить `Vary: Origin`.

### 4.6 Прекращение логирования токена

**Файл:** `server/cmd/vaultdb-server/main.go:206-214`

**Требование:** Не печатать полный токен. Сохранять в файл с правами 0600, выводить только hash-prefix.

### 4.7 Дефолтный токен в docker-compose

**Файл:** `docker-compose.yml:24`

**Требование:** Заменить `vdb_sk_local_dev` на:
```yaml
VAULTDB_API_TOKENS: "${VAULTDB_API_TOKENS:?Set VAULTDB_API_TOKENS in .env}"
```

### 4.8 Non-root в Dockerfile

**Файл:** `Dockerfile`

**Требование:** Добавить non-root user:
```dockerfile
# В builder stage:
RUN addgroup -S vaultdb && adduser -S vaultdb -G vaultdb
# В final stage:
COPY --from=builder /etc/passwd /etc/passwd
USER vaultdb
```

## Блок 5: Executor — исправление семантики (ВЫСОКИЙ)

### 5.1 DISTINCT до LIMIT

**Файл:** `server/internal/executor/commands_select.go:226-228`

**Проблема:** `DISTINCT` применяется после `LIMIT`.

**Требование:** Поменять порядок: сначала `distinctRows`, затем пагинация.

### 5.2 UPDATE ... FROM — корректные индексы

**Файл:** `server/internal/executor/commands_dml.go:491-500`

**Проблема:** Индексы кросс-продукта передаются в `UpdateRows`.

**Требование:** Сопоставить индексы из кросс-продукта с оригинальными индексами целевой таблицы.

### 5.3 UPSERT — уважение ON CONFLICT(columns)

**Файл:** `server/internal/executor/commands_dml.go:128-145`

**Проблема:** Сравниваются все колонки вместо указанных в `ON CONFLICT`.

**Требование:** Использовать `op.ConflictColumns` для определения колонок сравнения.

### 5.4 NTILE — защита от деления на ноль

**Файл:** `server/internal/executor/commands_select.go:1167-1172`

**Требование:** Проверять `n > 0`, иначе возвращать ошибку.

### 5.5 IN subquery — сравнение через CompareValues

**Файл:** `server/internal/executor/eval.go:808-818`

**Проблема:** Строгое строковое сравнение вместо типо-корректного.

**Требование:** Использовать `CompareValues` как в non-subquery пути.

### 5.6 ORDER BY — предвычисление ключей

**Файл:** `server/internal/executor/commands_select.go:684-707`

**Проблема:** `evalOperand` вызывается на каждое сравнение в сортировке.

**Требование:** Предвычислить ключи сортировки перед `sort.Slice`.

## Блок 6: Фронтенд (СРЕДНИЙ)

### 6.1 AbortController для всех fetch

**Файл:** `server/internal/httpserver/web/src/api/vaultdb.ts`

**Проблема:** Нет отмены запросов при unmount.

**Требование:**
1. Добавить `AbortController` в `parseResponse` или обёртку `fetchWithTimeout`.
2. Все хуки (`useQuery`, `useDatabases`) должны передавать `signal` и вызывать `abort()` в cleanup.

### 6.2 Runtime-валидация ответов API

**Файл:** `server/internal/httpserver/web/src/api/vaultdb.ts:59`

**Проблема:** `data as T` без проверки.

**Требование:** Добавить базовую проверку формы ответа (или использовать `zod`).

### 6.3 Токен — не localStorage

**Файл:** `server/internal/httpserver/web/src/App.tsx:14,24`

**Проблема:** Токен в `localStorage` — XSS-уязвимость.

**Требование:** Перенести в `sessionStorage` или cookie. Валидировать до сохранения.

### 6.4 Accessibility

**Файлы:** `QueryEditor.tsx`, `SchemaView.tsx`, `LoginScreen.tsx`, `DatabaseTree.tsx`

**Требование:** Добавить `aria-label` к: textarea, кнопке закрытия, инпуту пароля, навигации.

## Блок 7: Тестирование

### 7.1 Тесты для исправленных undo-функций

**Файл:** `server/internal/executor/commands_tx_test.go` (новый)

Тест-кейсы:
- INSERT → ROLLBACK: строки удалены, таблица пуста.
- UPDATE → ROLLBACK: старые значения восстановлены.
- DELETE → ROLLBACK: строки возвращены.
- Частичный apply с ошибкой → корректный откат applied ops.

### 7.2 Тесты для BufferPool

**Файл:** `server/internal/storage/buffer_pool_test.go` (новый)

Тест-кейсы:
- Fetch/Unpin корректно меняет pin count.
- Eviction корректно выбирает LRU-элемент.
- Все закреплённые страницы → evict возвращает false.
- Concurrent Fetch/Unpin.

### 7.3 Тесты для HeapFile Close race

**Файл:** `server/internal/storage/heap/heapfile_test.go`

Тест-кейсы:
- Concurrent ReadPage + Close: не должно быть паники.
- WritePage после Close: корректная ошибка.

### 7.4 Тесты для WAL Checkpoint

**Файл:** `server/internal/wal/wal_test.go`

Тест-кейсы:
- Checkpoint после записи → файл пуст/содержит только маркер.
- Simulated crash (truncate без sync) → recovery корректна.

## Верификация

1. **Unit-тесты:** `go test ./server/internal/...` — все существующие + новые тесты проходят.
2. **Интеграционные тесты:** `go test -tags=integration ./server/...` — полный цикл begin→DML→commit/rollback.
3. **Конкурентные тесты:** `-race` detector: `go test -race ./server/internal/storage/...`.
4. **Manual testing:** Запустить сервер, выполнить транзакции с ROLLBACK через CLI/веб, проверить data integrity.
5. **Security scan:** Проверить CORS, rate limiting, таймауты через `curl` и HTTP-клиент.
6. **Docker build:** `docker build .` и `docker-compose up` — контейнер запускается от non-root, дефолтный токен отсутствует.
