## Предисловие: что означает "Enterprise-first"

Это решение определяет всё остальное в документе, поэтому фиксируется явно
в самом начале.

VaultDB целится не в нишу "лёгкая встраиваемая альтернатива SQLite", а в нишу
"СУБД, которую ставит себе крупная организация вместо PostgreSQL/Oracle,
потому что VaultDB даёт то, чего у них нет": Time Travel из коробки, TDE
с envelope encryption и KMS, WAL с гарантированным crash recovery, честный
собственный протокол без чужих компромиссов.

Из этого следует:

- Мы не оптимизируем для минимального размера бинарника (SQLite-класса).
- Мы не жертвуем корректностью ради скорости выхода фич.
- Мы проектируем каждую подсистему так, будто её будут аудировать
  внешние security-инженеры и DBA с многолетним опытом на Oracle/PostgreSQL.
- Совместимость с существующей экосистемой (psql, ORM) — не цель.
  Цель — чтобы клиенты, которые ставят VaultDB, ставили её осознанно,
  за конкретные возможности, а не по инерции.

## 1. Протокол решений

Каждое решение зафиксировано в формате: Решение → Обоснование → Что это меняет.
Пересмотр решения возможен только через явный процесс (раздел 11), не через
молчаливое обхождение в коде.

---

### Решение 1 — WAL: сегментация

**Статус: ОТЛОЖЕНО. Синхронность — ДЕЛАЕМ.**

Сегментированный WAL (файлы по 16 МБ, ротация после checkpoint) не делается
до тех пор, пока метрики recovery time на реальных объёмах данных не покажут
проблему. Преждевременная оптимизация несуществующего узкого места.

Вместо этого реализуется `synchronous_commit` с тремя уровнями (`on`,
`off`, `remote_write`) — это даёт клиенту enterprise-грейд контроль над
durability/throughput trade-off уже сейчас, стоит на порядок дешевле.

**Что это меняет:** WAL остаётся одним линейно растущим файлом с checkpoint
и truncate до текущей реализации. Задача сегментации переносится в Фазу 2
как часть работы над WAL нового уровня, с чётким триггером активации:
recovery time > 30 секунд на реальных данных клиента, либо WAL archiving
для PITR становится конкретным запросом заказчика.

---

### Решение 2 — Каталог: консистентность

**Статус: ДВУХШАГОВОЕ. Rename — сейчас. Бинарный каталог — часть Фазы 2.**

Немедленно: JSON-каталог переводится на write-to-temp + `os.Rename()`.
Это устраняет половину риска рассинхронизации почти бесплатно.

Полное решение (бинарный каталог на dedicated pages внутри page engine,
атомарность через WAL) не выделяется в отдельный проект — оно сливается
с уже идущим `storage-engine-rewrite-plan.md`. Два параллельных рефакторинга
одной и той же подсистемы хранения — трата ресурсов и риск конфликтов.

**Что это меняет:** отдельного трека "переписать каталог" не существует.
Каталог мигрирует на page-based хранение синхронно с миграцией
пользовательских таблиц, по единому плану storage engine rewrite.

---

### Решение 3 — Тестирование: закрыть дыры немедленно

**Статус: ПРИНЯТО ПОЛНОСТЬЮ. Приоритет выше любой фичи из этого документа.**

Это единственный раздел исходного документа, который не требует
взвешивания trade-off — все четыре пункта дёшевы и находят реальные баги.

- Fuzz-тестирование парсера (`go test -fuzz`) — внедряется в CI как
  постоянно работающий job, не разовая проверка.
- `go test -race -count=1000` на существующих stress-тестах — становится
  обязательным gate перед мержем в `main` для любого PR, трогающего
  concurrency-код (executor, txmanager, bufpool, wal).
- Генератор случайных валидных SQL-запросов (аналог SQLsmith) — реализуется
  как отдельная тестовая утилита, гоняется в nightly CI.
- Regression benchmark suite — каждый PR, который может повлиять на
  производительность, обязан прогонять существующие бенчмарки и приложить
  сравнение с `main`. Автоматизируется через `benchstat`.

**Что это меняет:** Enterprise-позиционирование требует, чтобы клиент
доверял стабильности ядра больше, чем скорости появления новых фич.
Тестирование — не отдельная фаза, а постоянный процесс с первого дня.

---

### Решение 4 — Протокол: собственный, без PostgreSQL wire compatibility

**Статус: ПРИНЯТО. Подтверждено пользователем явно.**

VaultDB не реализует PostgreSQL wire protocol v3. Причины окончательные:

1. Time Travel, TDE-метаданные, AI-функции и будущие фичи VaultDB не
   вписываются в протокол, спроектированный под другую СУБД — попытка
   впихнуть их создаёт постоянно растущий список "почти совместимых, но
   не совсем" углов, что хуже честного отдельного протокола.
2. Экосистема PostgreSQL (драйверы, ORM, pgBouncer) оптимизирована под
   поведение PostgreSQL, а не VaultDB — совместимость на уровне протокола
   не даёт совместимости на уровне поведения, только иллюзию.
3. Ресурсы команды направляются на то, что уникально отличает VaultDB,
   а не на копирование чужого протокола.

Взамен — собственный протокол v2, спроектированный с нуля под нужды
VaultDB, полностью описан в Фазе 1 этого документа.

**Что это меняет:** весь трек "PostgreSQL compatibility" закрыт. Ресурсы,
которые могли пойти на это, перенаправляются на Фазу 1 (протокол v2) и
Фазу 5 (SQL compatibility по существу, а не по wire-формату).

---

### Решение 5 — Модель расширений: WASM UDF

**Статус: ПРИНЯТО К РАЗРАБОТКЕ, но не первым приоритетом.**

WASM-расширения (через `wazero`, Go-native рантайм без CGo) — это
правильное enterprise-решение: sandbox, любой язык компиляции, безопасность
по дизайну. Это делается, но после того как ядро (протокол, каталог,
security) стабилизировано.

Синтаксис `CREATE FUNCTION` проектируется уже сейчас с учётом будущего
WASM-тела, чтобы при реализации не потребовался breaking change:

```sql
-- Уже сегодня — SQL-тело:
CREATE FUNCTION calc_discount(price FLOAT) RETURNS FLOAT
LANGUAGE SQL
AS 'SELECT price * 0.9';

-- Синтаксис заранее расширяем для будущего WASM (Фаза 6):
CREATE FUNCTION hash_pii(value TEXT) RETURNS TEXT
LANGUAGE WASM
AS 'file:///plugins/hash_pii.wasm'
WITH (memory_limit = '16MB', timeout = '100ms');
```

**Что это меняет:** грамматика `CREATE FUNCTION` расширяется под
`LANGUAGE WASM` заранее (Фаза 5), сама реализация рантайма — Фаза 6.

---

### Решение 6 — Runtime и память

**Статус: ПРИНЯТО. sync.Pool — сейчас. Arena — наблюдение.**

`sync.Pool` для горячих структур (`storage.Row`, буферы сериализации,
`[]Expression`) внедряется в рамках Фазы 4. Дёшево, обратимо, измеримый
эффект на GC pressure.

Arena allocator (`arena.Arena`) — не внедряется пока API находится под
экспериментальным флагом Go. Enterprise-СУБД не может зависеть от
нестабильного API языка. Решение пересматривается при стабилизации API
в релизе Go (отслеживается как внешняя зависимость, не внутренняя задача).

Дополнительно к исходному документу: вводится документированная политика
`GOGC` и рекомендации по `GOMEMLIMIT` для рабочих наборов данных 10GB+ —
это часть Enterprise Deployment Guide (Фаза 4).

---

### Решение 7 — Security сверх шифрования

**Статус: SQL injection аудит — немедленно. Audit log — Фаза 3, с tamper-evidence.**

SQL injection аудит выполняется вне очереди, до начала любой другой работы
из этого документа. Обоснование: разрыв между уровнем защиты данных
(AES-256-GCM, envelope encryption, KMS) и потенциальной дырой в
`PREPARE`/`CREATE FUNCTION`/`CALL` — это не просто баг, это репутационная
катастрофа при обнаружении внешним аудитором. Проверяются все места, где
`parser.Value.StrVal` участвует в конструкциях, повторно парсящихся
(функции, триггеры, миграции, EXECUTE с параметрами).

Audit log реализуется в Фазе 3 как встроенная системная таблица
(`vaultdb_audit_log`), не внешний лог-файл — потому что enterprise-заказчики
должны иметь возможность запросить журнал аудита через SQL, встроить в
свои SIEM-системы через тот же протокол, наложить RLS-политики на сам
audit log.

Дополнение, отсутствовавшее в исходном документе: audit log проектируется
как append-only с hash-chain — каждая запись содержит хеш предыдущей
записи. Без этого свойства audit log — это обычный журнал, который можно
незаметно отредактировать, и он не выдерживает никакой формальной
compliance-проверки (SOC 2, ISO 27001).

Embedded mode без auth (пункт B исходного документа) — документируется
как есть, отдельной работы не требует, поскольку embedded — не основное
позиционирование продукта (см. Решение 8).

### Решение 8 — Упаковка и распространение

**Статус: Shared library — не делается. Docker остаётся основным каналом.**

Прямое следствие Enterprise-first позиционирования. C-совместимая shared
library (`.so`/`.dylib`/`.dll`) имеет смысл только для embedded-стратегии —
а VaultDB не позиционируется как замена SQLite внутри процесса. Компании,
которым нужны TDE, KMS, Time Travel и WAL crash recovery, разворачивают
VaultDB как сервис, а не линкуют библиотеку в свой бинарник.

Размер бинарника (~15МБ) не оптимизируется агрессивно (UPX и подобное) —
это не критерий для enterprise-деплоя, где счёт идёт не на килобайты,
а на гарантии надёжности.

Docker остаётся основным каналом распространения, с усилением в сторону
enterprise-паттернов деплоя: Kubernetes Helm chart, поддержка readiness/
liveness probes через существующий `/health`, документированные ресурсные
профили (small/medium/large deployment sizing).

**Что это меняет:** задача "shared library для embedded" удаляется из
бэклога полностью, не откладывается — это осознанное "нет", а не "потом".

### Решение 9 — SQL Compatibility Roadmap

**Статус: ПРИНЯТО с уточнением приоритетов.**

Согласован предложенный core-набор для полноты SQL-движка:

| Фича | Статус решения |
|---|---|
| EXPLAIN ANALYZE | Фаза 5, высокий приоритет (developer UX уже требуется существующими клиентами) |
| COPY FROM/TO | Фаза 5, высокий приоритет (ETL — блокер для enterprise-миграций с других СУБД) |
| DEFAULT expressions (CURRENT_TIMESTAMP, gen_random_uuid) | Фаза 5 |
| GENERATED ALWAYS AS ... STORED | Фаза 5 |
| DISTINCT ON | Фаза 5, низкая стоимость |
| JSONB операторы (->, ->>, @>, ?) | Фаза 5, после core-набора |
| Array types (INT[], TEXT[]) | Фаза 5, после core-набора |
| PARTITION BY RANGE/LIST/HASH | Фаза 6+, требует зрелого storage engine |
| Full-text search enterprise-уровня | Фаза 6+ |
| INSERT ... ON DUPLICATE KEY (MySQL-совместимость) | Не приоритет — конфликтует с решением не гнаться за чужой совместимостью. UPSERT/MERGE уже покрывает эту потребность |

Дополнение к исходному документу: явно зафиксировано, что тестовое
покрытие комбинации RETURNING + UPSERT/MERGE добавляется как часть Фазы 5,
поскольку это типичная граница, где скрываются баги в уже реализованных
фичах.

## 2. Архитектура фаз

Фазы упорядочены по зависимостям, не по календарному времени — сроки
принципиально не фиксируются согласно требованию. Каждая фаза может
занять произвольное время; переход к следующей происходит по готовности,
подтверждённой чеклистом приёмки этой фазы.

```
✅ Фаза 0 — Фундамент корректности — ВЫПОЛНЕНА
✅ Фаза 1 — Собственный протокол v2 — ВЫПОЛНЕНА
✅ Фаза 2 — Каталог и WAL нового уровня — ВЫПОЛНЕНА
✅ Фаза 3 — Security и Audit — ВЫПОЛНЕНА
✅ Фаза 4 — Runtime и производительность — ВЫПОЛНЕНА
✅ Фаза 5 — SQL Compatibility — ВЫПОЛНЕНА
✅ Фаза 6 — WASM UDF + Partitioning + FTS — ВЫПОЛНЕНА
         |
         v
🔄 Фаза 7 — Hardening и Stabilization — В ПРОЦЕССЕ
   (crash-тесты: 60%, benchmark: 75%, edge cases: 25%, security: 60%, docs: 30%)
```

Фазы 0 и 3 (SQL injection, тестирование) не блокируют друг друга и
выполняются параллельно с самого начала — это не последовательный этап,
а постоянно активный процесс.

## 3. Фаза 0 — Фундамент корректности

Цель: устранить структурные риски прежде чем наращивать функциональность.
Это не "фича", это гигиена, без которой Enterprise-заявление несостоятельно.

### 3.1 SQL Injection Audit

Объём проверки:
- `parser.Value.StrVal` во всех местах, где строка повторно парсится:
  PREPARE / EXECUTE, CREATE FUNCTION body, CREATE PROCEDURE body,
  CALL с аргументами, миграции, триггеры (если есть)
- Любая конкатенация пользовательского ввода в SQL-текст перед повторным Parse()
- Проверка что параметризация ($1, $2...) в PREPARE не подвержена
  экранированию через контроль типов на этапе bind, а не текстовую подстановку

Результат аудита оформляется как отдельный security-отчёт с перечнем
найденных мест и статусом исправления каждого. Отчёт хранится в репозитории
(`docs/security/sql-injection-audit-YYYY-MM-DD.md`) как постоянный
compliance-артефакт.

### 3.2 Fuzz-тестирование парсера — постоянный CI-процесс

```go
// internal/parser/fuzz_test.go

func FuzzParse(f *testing.F) {
    // Seed corpus — известные валидные и граничные случаи
    f.Add("SELECT * FROM users;")
    f.Add("")
    f.Add("\x00\x00\x00")
    f.Add(strings.Repeat("(", 10000))
    f.Add("SELECT " + strings.Repeat("a,", 50000) + "b FROM t;")

    f.Fuzz(func(t *testing.T, input string) {
        defer func() {
            if r := recover(); r != nil {
                t.Fatalf("parser panicked on input %q: %v", input, r)
            }
        }()
        _, _ = Parse(input) // не паникует — единственное требование
    })
}
```

```yaml
# CI: nightly fuzz job, не блокирует PR, но алертит команду
fuzz-nightly:
  schedule: "0 2 * * *"
  run: go test -fuzz=FuzzParse -fuzztime=2h ./internal/parser/
```

### 3.3 Race-тесты как обязательный gate

```yaml
# .github/workflows/ci.yml — добавить обязательную проверку
race-detector:
  run: go test -race -count=1000 ./internal/executor/... ./internal/txmanager/... ./internal/storage/...
  # Обязателен для PR, затрагивающих: executor, txmanager, bufpool, wal, broadcaster
```

### 3.4 SQL-генератор случайных запросов (аналог SQLsmith)

```go
// tools/sqlfuzz/generator.go

// RandomQueryGenerator генерирует случайные валидные SQL-запросы
// на основе существующей схемы БД — для проверки что сервер не падает
// на неожиданных, но синтаксически корректных комбинациях.
type RandomQueryGenerator struct {
    schema *catalog.Schema
    rng    *rand.Rand
}

func (g *RandomQueryGenerator) GenerateSelect() string {
    // Случайные комбинации: JOIN N таблиц, случайные WHERE-условия,
    // случайные оконные функции, случайная вложенность подзапросов
    return ""
}

// nightly job: 10000 случайных запросов, единственная проверка — сервер жив
```

### 3.5 Regression Benchmark Suite

```bash
# tools/benchstat-gate.sh — сравнение бенчмарков PR vs main
go test -bench=. -benchmem -count=10 ./... > new.txt
git stash && go test -bench=. -benchmem -count=10 ./... > old.txt && git stash pop
benchstat old.txt new.txt
# Если регрессия > 10% на ключевых бенчмарках — CI помечает PR предупреждением
```

## 4. Фаза 1 — Собственный протокол v2

Цель: протокол, спроектированный под VaultDB с нуля, документированный
как формальная спецификация, с официальными клиентскими библиотеками.

### 4.1 Почему текущий протокол недостаточен для Enterprise

Текущий JSON-over-TCP работает, но не formalized: нет версионирования
протокола, нет официальной спецификации вне кода, нет generated-клиентов.
Enterprise-заказчик не подключает СУБД, у которой протокол — это "то, что
делает код", а не документированный контракт.

### 4.2 Протокол v2 — требования

1. Версионирование протокола — Handshake сообщает версию, клиент и сервер
   могут договориться о совместимой версии.

2. Формальная спецификация в отдельной директории:
   `docs/protocol/vaultdb-protocol-v2.md` — независимо от реализации,
   так что сторонние разработчики могут писать клиентов без чтения Go-кода.

3. Расширяемость под уникальные фичи VaultDB:
   - Time Travel параметры как first-class часть запроса, не хак поверх SQL
   - Метаданные шифрования (какой DEK version использован) в ответе
   - AI-режимы (если сохраняются) как отдельный тип сообщения, не оверлей на SQL

4. Бинарный режим для горячего пути (опционально, поверх текущего NDJSON):
   Protocol Buffers или собственный компактный бинарный формат
   для клиентов, которым важна пропускная способность.
   NDJSON остаётся default — для отладки, простоты, HTTP-совместимости.

5. Официальные клиентские библиотеки:
   - Go (эталонная реализация, используется в vaultdb-cli)
   - Python
   - JavaScript/TypeScript (Node + браузер через WASM-клиент)

## 6. Фаза 3 — Security и Audit

### 6.1 Audit Log — спецификация

```sql
-- Системная таблица, доступна через обычный SQL (с ограничением прав)
SELECT * FROM vaultdb_audit_log
WHERE actor = 'admin@company.com'
  AND action = 'ALTER TABLE'
ORDER BY occurred_at DESC
LIMIT 50;
```

```go
// internal/audit/log.go

// AuditEntry — одна запись журнала аудита.
// Append-only, hash-chained — каждая запись содержит хеш предыдущей.
type AuditEntry struct {
    ID          uint64
    OccurredAt  time.Time
    Actor       string
    Action      string
    Target      string
    Details     string
    PrevHash    string
    EntryHash   string
}

// Append добавляет запись и поддерживает hash-chain.
func (a *AuditLog) Append(entry AuditEntry) error {
    prevHash, _ := a.lastHash()
    entry.PrevHash = prevHash

    payload := fmt.Sprintf("%d|%s|%s|%s|%s|%s",
        entry.ID, entry.OccurredAt.Format(time.RFC3339),
        entry.Actor, entry.Action, entry.Target, entry.PrevHash)
    hash := sha256.Sum256([]byte(payload))
    entry.EntryHash = hex.EncodeToString(hash[:])

    return a.storage.InsertRows(ctx, "system", "vaultdb_audit_log", []Row{entry.ToRow()})
}

// VerifyChain проверяет целостность всей цепочки — обнаруживает подмену записей.
func (a *AuditLog) VerifyChain() (bool, error) {
    entries, _ := a.storage.ReadCurrentRows(ctx, "system", "vaultdb_audit_log")
    prevHash := ""
    for _, e := range entries {
        expected := computeHash(e, prevHash)
        if expected != e.EntryHash {
            return false, fmt.Errorf("audit chain broken at entry %d", e.ID)
        }
        prevHash = e.EntryHash
    }
    return true, nil
}
```

```sql
-- Служебная команда проверки целостности аудита (для compliance-отчётов)
VERIFY AUDIT LOG;
-- Output: "Audit chain intact: 48392 entries verified, no tampering detected."
```

### 6.2 Что логируется

| CREATE/DROP DATABASE | Да |
| CREATE/ALTER/DROP TABLE | Да |
| CREATE/DROP INDEX | Да |
| Изменение прав доступа (RLS-политики) | Да |

## 9. Фаза 6 — Расширяемость и enterprise-масштаб

### 9.1 WASM UDF

Реализация через wazero (Go-native WASM runtime, без CGo — важно для
сохранения простоты сборки, соответствует Enterprise-требованию
предсказуемой поставки).

```go
// internal/wasmudf/runtime.go

type WASMFunction struct {
    Name         string
    ModulePath   string
    MemoryLimit  uint32
    Timeout      time.Duration
    runtime      wazero.Runtime
    compiled     wazero.CompiledModule
}

func (f *WASMFunction) Call(ctx context.Context, args []storage.Value) (storage.Value, error) {
    ctx, cancel := context.WithTimeout(ctx, f.Timeout)
    defer cancel()

    mod, err := f.runtime.InstantiateModule(ctx, f.compiled, wazero.NewModuleConfig())
    if err != nil { return nil, fmt.Errorf("wasm instantiate: %w", err) }
    defer mod.Close(ctx)

    result, err := mod.ExportedFunction("execute").Call(ctx, marshalArgs(args)...)
    if err != nil { return nil, fmt.Errorf("wasm execute (timeout=%s): %w", f.Timeout, err) }

    return unmarshalResult(result), nil
}
```

### 9.2 Partitioning (RANGE/LIST/HASH)

Требует зрелого page engine (Фаза 2 завершена) — партиция физически
является отдельным набором heap-файлов с общим каталожным описанием.
Детальная спецификация пишется отдельным документом при подходе к этой
фазе, когда storage engine стабилизируется.

### 9.3 Full-Text Search enterprise-уровня

Расширение существующего FTS до уровня ranking-алгоритмов (BM25),
поддержки нескольких языков стемминга, подсветки сниппетов —
конкурентно с Elasticsearch для случаев, где не нужен отдельный
поисковый кластер.

## 10. Фаза 7 — Hardening и Stabilization

Цель: стабилизировать то, что уже построено. Не новые фичи — а доведение
существующего до production-ready уровня.

### 10.1 Crash-тесты

- Увеличить покрытие crash-сценариев: kill -9 в момент DDL, DML, checkpoint
- Проверить recovery после каждого типа операции
- Добавить chaos-тесты с конкурентными операциями
- Критерий: 100% crash-сценариев восстанавливаются без потери данных

### 10.2 Benchmark Regression Gate

- Запускать benchmarks на каждом PR, затрагивающем executor/storage/wal
- Сравнивать с baseline на main
- Регрессия > 10% — блокирует мерж
- Критерий: история бенчмарков за 20+ PR, ни одной незамеченной регрессии

### 10.3 Edge Cases

- Тестирование на больших данных (1M+ строк)
- Concurrent mixed workloads (OLTP + DDL одновременно)
- Unicode/emoji во всех текстовых операциях
- Пустые таблицы, NULL значения, boundary values
- Критерий: нет паник и data corruption за 24 часа непрерывной нагрузки

### 10.4 Security Hardening

- Повторный SQL injection audit после всех изменений
- Fuzz-тесты парсера: 2+ часа nightly без паник
- Race-тесты: -race -count=1000 на всех concurrency-пакетах
- Критерий: 0 known vulnerabilities, 0 race conditions

### 10.5 Documentation Hardening

- Проверить все примеры кода — они должны работать
- Проверить все SQL-запросы из документации — они должны выполняться
- Убедиться что API reference актуален
- Критерий: любой разработчик может запустить проект по документации за 15 минут

## 12. Управление документом

Любое решение, зафиксированное в разделе 1, может быть пересмотрено только
через явный процесс:

1. Инициатор формулирует: какое решение пересматривается и почему
   (новые данные, изменившиеся требования заказчика, техническое открытие)
2. Решение обновляется в этом документе с пометкой:
   "Решение N пересмотрено [дата]: было — X, стало — Y. Причина: ..."
3. Старое обоснование не удаляется — сохраняется в истории для контекста

Документ не переписывается тихо. История решений — часть enterprise-грейд
дисциплины разработки, которую сам продукт (Time Travel, Audit Log)
проповедует для пользовательских данных — было бы странно не следовать
тому же принципу для собственных архитектурных решений.
